## What this app is

`furniture-id-app` is a professional-grade antique furniture identification web
app. The owner-developer is building it for resale to professional appraisers,
estate liquidators, and antique dealers. **Brand promise: "the tool that
professionals use — complete and thorough."** That's the target bar.

Architecture (high level):
- **P0** = Claude Sonnet vision call. Captures observations as structured JSON
  (`clue`, `category`, `confidence`, `description` prose).
- **P1–P7** = deterministic phases in `lib/engine.ts` that consume P0 output to
  produce form identification, dating, style attribution, valuation, and the
  user-facing report.
- Hosted on Render. Repo restricted; GitHub access via `mcp__github__*` tools.

## The single most important architectural insight (settled, owner-confirmed)

> **The engine has the answers in its own code — captured by P0 in observation
> descriptions and authored in canonical reference data — but throws them away
> at every consumption point and computes answers from a thin slice of metadata
> instead.**

Owner's words: *"It sees it, stores it, has an opportunity to use it, but
throws it away instead."*

Concretely:
- **P0 captures rich appraiser-voice prose** in every `observation.description`
  — explicit dates ("c. 1920–1940"), period attributions ("Verner Panton's Cone
  Chair, 1958"), construction reasoning, maker names verbatim.
- **The reference data the owner authored is comprehensive** —
  `lib/constraints/hardware.ts`, `woodEvidence.ts`, `joinery.ts`,
  `makerMarks.ts`, etc. Each canonical entry has `period_associations`,
  `false_positive_warnings`, `identifying_characteristics`, `indicator_text`,
  `replacement_likelihood`, `regional_context`, multi-line detail.
- **The engine reads almost none of it.** The dating layer consults a one-line
  `dateHint` per clue. The form selector substring-matches against keywords. The
  style attributor scrapes tokens. Display assembly ignores both observation
  descriptions AND the rich reference fields.

When the engine's arithmetic on the thin slice contradicts what the LLM wrote,
the reconciliation layer either picks one or invents a label to bridge the gap.
That produces outputs like *"Art Deco vocabulary (post-1870 reproduction)
Dresser, c. 1870–1910"* (S052 — reproduction before the style existed) and
*"Grandmillennial Colonial Secretary desk, c. 1990–2100"* (S054 — date
projected 74 years into the future).

**The fix is not smarter rules, not more scoring math, not new architecture. The
fix is making the engine consult what's already there.** At every consumption
point that reads only the thin slice, widen the read to include the rich
evidence and the rich reference.

## The next concrete fix to ship: date-prose extraction wired into convergence

1. **Extractor.** Read every observation description for explicit date phrases:
   `c. YYYY–YYYY`, `c. YYYY`, `post-YYYY`, `pre-YYYY`, `YYYYs–YYYYs`, and
   named-period anchors (Victorian era, Art Deco era, Federal period…) backed by
   a small lookup table.
2. **Aggregate** extracted ranges across all observations on a scan. Intersect
   overlapping ranges → the "LLM-prose-derived date envelope."
3. **Wire into convergence.** Either add a `prose` layer in
   `engineDatingOverlap.ts`'s `CATEGORY_TO_LAYER`, OR add a parallel input to
   `buildDatingOverlap` alongside `styleAttribution`. Authority weighting:
   moderate — above style attribution alone, below physical-evidence layers
   (joinery, toolmarks).

Closes ~11 corpus specimens sharing the shape "LLM said X in prose, engine
computed not-X in date math": S011, S012, S015, S016, S017, S038, S045, S048,
S052, S054, S025.

**Scope honesty given to the owner:** 3–5 days end-to-end including corpus
validation. Not 17 days. Not 17 weeks.

**Architectural blocker (goes with step 3):** `CATEGORY_TO_LAYER`
(`lib/engineDatingOverlap.ts:493`) skips `style` and `construction` categories.
Most date-bearing clues fall there. This is `docs/stress-test-findings.md` #18
("Construction/mechanism evidence excluded from dating — DEFERRED ENGINE TASK").
The deferral is now an active blocker for the work the owner wants done.

**After the date-prose wire:** same template applies to style attribution (read
prose, not just keyword tokens — closes most M11 specimens) and to form
selection (read `*_form` clue descriptions, not just substring-match against
form aliases — closes most M8 specimens including the LOOM MISID family and the
`form_windsor_chair` spindle-synthesis cases).

**Father's Day target: June 21, 2026** — soft/beta launch for the owner's own
community / dealer network, not a broad public launch.
